Detailed Description

template<class C, class RE = region_environment<pts_t,pts_t,pts_t,pts_t,pts_t>>
class shyft::core::region_model< C, RE >

region_model is the calculation model for a region, where we can have one or more catchments. The role of the region_model is to describe region, so that we can run the region computational model efficiently for a number of type of cells, interpolation and catchment level algorihtms.

The region model keeps a list of cells, of specified type C

Template Parameters
C
See also
cell type parameter, describes the distributed cell calculation stack.
Template Parameters
REregion environment class The cell-type must provide the following types:
  1. parameter_t parameter type for the cell calculation stack
  2. state_t state type for the cell
  3. response_t response type for cell
  4. timeaxis_t timeaxis type for the cell environment input/result ts. and following members/methods must be available
  5. .run(time_axis) executes the method stack for the cell, (must be thread-safe!)
  6. .env_ts with env. parameters

Initializes the cell enviroment (cell.env.ts* )

The initializes the cell environment, that keeps temperature, precipitation etc that is local to the cell. The intial values of these time-series is set to zero. The region-model time-axis is set to the supplied time-axis, so that the any calculation steps will use the supplied time-axis. This call is needed prior to call to interpolate() or run_cells(). The call ensures that all cells.env ts are reset to zero, with a time-axis and value-vectors according to the supplied time_axis. Also note that the region_model.time_axis is set to the supplied time-axis.

void shyft::core::region_model< C, RE >::run_cells ( size_t  use_ncore = 0,

run_cells calculations over specified time_axis the cell method stack is invoked for all the cells, using multicore up to a maximum number of tasks/cores. Notice that this implies that executing the cell method stack should have no side-effects except for invocation of the cell state_collector, response_collector. No thread should run simultaneously on the same cell, and the collectors must be carefully written to avoid any race-condition or memory contention (lock-free code).

Precondition
cells should be prepared by prior calls so that
  1. all cells have env_ts intialized with env. precip,temp,rad etc.
  2. all cells have state set to the initial values
  3. catchment_parameters should be set (and applied to the matching cells)
Postcondition
:
  1. all cells have updated state variables
Parameters
use_ncoreif 0 figure out threads using hardware info, otherwise use supplied value (throws if >100x ncore)
start_stepof time-axis, defaults to 0, starting at the beginning
n_stepsnumber of steps to run from the start_step, a value of 0 means running all time-steps
Returns
void

inline

initializes cell.env and project region env. time_series to cells.

Note
Prior to running all cell.env_ts.xxx are reset to zero, and have a length of time_axis.size().

Only supplied vectors of temp, precip etc. are interpolated, thus the user of the class can choose to put in place distributed series in stead.

This call simply calls initialize_cell_environment() followed by interpolate<..>(..)

Parameters
ip_parametercontains wanted parameters for the interpolation
time_axisshould be equal to the shyft::time_axis the region_model is prepared running for.
envcontains the region_environment type
best_effort(default=true)
Returns
true if entire process was done with no exceptions raised
